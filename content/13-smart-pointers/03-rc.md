<h1 id="为什么需要多所有权">为什么需要多所有权？</h1>
<p>Rust 的所有权规则确保每个值都有唯一的所有者，这是内存安全的基础。但在某些场景下，这条规则会成为障碍。</p>
<p>设想我们要构建这样一个图结构：两个列表 <code>b</code> 和 <code>c</code> 需要共享同一段数据 <code>a</code>。如果我们用 <code>Box<T></code> 来实现，会遇到什么问题？</p>

<p><code>Box<T></code> 独占所有权，一旦 <code>a</code> 被移入 <code>b</code>，就无法再被 <code>c</code> 使用。我们可以改用引用 <code>&List</code>，但这会带来复杂的生命周期标注。在这种场景下，<code>Rc<T></code> 才是正确的选择。</p>
<h1 id="rct-的工作原理"><code>Rc<T></code> 的工作原理</h1>
<p><code>Rc<T></code> 是 <strong class="strong-star">Reference Counting</strong>（引用计数）的缩写。它在堆上存储数据，同时维护一个计数器，记录当前有多少个地方持有这份数据的引用。</p>
<ul>
<li>每次调用 <code>Rc::clone</code>，计数器加 1</li>
<li>每当一个 <code>Rc<T></code> 值离开作用域，计数器减 1</li>
<li>当计数器归零时，数据才会被真正释放</li>
</ul>

<h2 id="为什么用-rcclonea-而不是-aclone">为什么用 <code>Rc::clone(&a)</code> 而不是 <code>a.clone()</code>？</h2>
<p>在技术上，两种写法都能编译。但 Rust 社区<strong class="strong-star">强烈推荐</strong>函数式写法 <code>Rc::clone(&a)</code>，原因在于<strong class="strong-star">语义上的清晰区分</strong>：</p>
<ul>
<li><code>deep_clone()</code> 或 <code>.clone()</code>：通常意味着对数据的<strong class="strong-star">深拷贝</strong>，可能非常耗时（比如拷贝整个向量）。</li>
<li><code>Rc::clone(&a)</code>：仅仅是<strong class="strong-star">增加一个计数器的值</strong>，无论数据有多大，这个操作都是 O(1) 的轻量级操作。</li>
</ul>
<p>当你阅读代码时，看到 <code>Rc::clone</code> 可以立刻判断”这里没有性能问题”；而看到 <code>.clone()</code> 则需要停下来思考是否会触发深拷贝。</p>
<h1 id="rct-的限制"><code>Rc<T></code> 的限制</h1>
<p><code>Rc<T></code> 的设计做了一个重要取舍：</p>
<p><strong class="strong-star">它只提供不可变引用。</strong> 在任何时候，你只能对 <code>Rc<T></code> 内部的数据进行只读访问。</p>
<p>这是必要的安全保障：如果允许多个所有者同时进行可变修改，就会产生数据竞争（即使在单线程中也会导致逻辑混乱）。</p>
<p>此外，<code>Rc<T></code> <strong class="strong-star">只能用于单线程</strong>。它的引用计数操作不是原子性的，无法在线程之间安全地共享。如果你需要在多线程中共享数据，应该使用它的并发版本 <code>Arc<T></code> (Atomic Reference Counting)，这将在<a href="/RustCourse/chapters/14-concurrency/00-index">并发编程</a>章节中介绍。</p>
<p>如果你既需要多所有权，又需要可变性，就需要将 <code>Rc<T></code> 与下一节介绍的 <code>RefCell<T></code> 组合使用。</p>
